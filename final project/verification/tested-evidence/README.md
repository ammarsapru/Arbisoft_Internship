# Test-suite classification and evidence

Waypoint has three distinct deterministic test layers plus opt-in live-model evaluation.
The unit/integration distinction is executable, not only documentation: `conftest.py`
assigns exactly one marker to every backend test module and fails collection if a new test
module is not classified.

## Verified suite inventory — July 24, 2026

| Layer | Count | External model/network required | Purpose |
|---|---:|---|---|
| Unit | 40 | No | Isolated algorithms, parsers, policies, adapters, and validation behavior |
| Integration | 33 | No | Cross-component graph/index/session/API/MCP/agent behavior |
| Backend total | 73 | No | Whole-backend and core-feature coverage inputs |
| HTTP E2E | 1 Flask workflow | Repository checkout only | Real running HTTP server through analysis and navigation endpoints |
| Browser E2E | Playwright scenarios | No live model in deterministic suite | UI-to-HTTP behavior, layout, chat, evidence, and navigation |
| Live model | Opt-in | Yes | Provider behavior, multi-step agent execution, and model comparison |

## Unit tests — 40 verified tests

Unit tests isolate one algorithm or service boundary. They use temporary files and mocks,
do not start an HTTP server, and do not call live model providers.

| Module | Main responsibility tested |
|---|---|
| `test_analyzer.py` | Python AST definitions, imports, calls, evidence, path safety, limits |
| `test_polyglot_analyzer.py` | Java/JS/TS/TSX/HTML/CSS parsing and relationship resolution |
| `test_provider.py` | Provider request translation, usage mapping, model routing, errors |
| `test_questions.py` | Deterministic question construction and scoring |
| `test_repository_import.py` | GitHub URL policy, clone limits, checkout accounting, errors |
| `test_observability.py` | Structured tracing, value bounding, redaction, hook output |
| `test_issues.py` | Issue data transformation and deterministic issue behavior |

Run only unit tests:

```powershell
.\.venv\Scripts\python.exe -m pytest -m unit --no-cov
```

Verified result: **40 passed, 33 deselected**.

| Module | Boundaries crossed |

| `test_model_comparison.py` | frozen retrieval → two model fixtures → independent validation and metering |

## Coverage verification

Coverage is intentionally generated by the complete 73-test deterministic backend suite,
because both isolated logic and cross-component branches are part of the executed product.

```powershell
# Whole backend: fails below 70%.
.\.venv\Scripts\python.exe -m pytest

# Every declared core feature and the weighted core aggregate must each be >=70%.
.\.venv\Scripts\python.exe -m verification.scripts.core_coverage_report

# Unified readable whole-backend + core-feature report.
.\.venv\Scripts\python.exe -m verification.scripts.coverage_report
```

The explicit core module manifest is `verification/core-features.json`; the unified result
is `verification/results/coverage-report.md`.

## HTTP end-to-end test

`verification.scripts.http_flask_e2e` sends real requests to a running backend and verifies:

1. Flask analysis/import;
2. persistent analysis identity;
3. summary and index status;
4. bounded source reads;
5. architecture report;
6. symbol usage;
7. graph neighborhood traversal.

This is E2E rather than a unit/integration fixture because it crosses the process-level HTTP
boundary and uses a real moderate repository.

## Browser end-to-end tests

Playwright scenarios live in `frontend/e2e`. The deterministic browser suite mocks model
endpoints so UI correctness does not depend on credits, while repository analysis and HTTP
integration remain real.

```powershell
npm.cmd --prefix frontend run test:e2e
```

## Live-model separation

Live provider evaluation is deliberately not included in coverage. Coverage answers whether
our code executed; live evaluation answers whether an external model followed tools, schema,
and evidence constraints. Mixing them would make CI nondeterministic and make coverage depend
on quota.
