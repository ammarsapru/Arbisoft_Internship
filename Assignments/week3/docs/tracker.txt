EVAL RESULTS TRACKER
========================
All numbers below are from real runs against the local Ollama server
(embedding models: qwen3-embedding:0.6b, snowflake-arctic-embed:latest;
chat model: qwen2.5:1.5b). Nothing here is simulated or estimated.

Four generations of results are recorded, each a real, independent run:
  RUN A (2026-07-07, single-blob prompt) -- one plain string passed to
    ChatOllama.invoke(), persona + rules + context + question all mashed
    into a single user turn (PROMPT_TEMPLATE, since removed).
  RUN B (2026-07-07, system+user prompt split) -- SystemMessage (persona,
    capabilities, hard constraints, output contract) + HumanMessage
    (RETRIEVED CONTEXT with per-chunk relevance scores, distinct from a
    USER QUESTION section). Grounding checks: citation_pages,
    quotes_grounded, numbers_grounded only.
  RUN C (2026-07-08, citations-required + unconditional auto-reject) --
    added hallucination.check_citations_required (an answer claiming
    sufficient context must cite something) and a self-check step that
    deterministically discarded any answer with zero citations, no retry.
    KEPT HERE AS A CAUTIONARY DATA POINT, NOT THE SHIPPED DESIGN -- see
    section 4.
  RUN D (2026-07-08, bounded retry-with-feedback loop) -- CURRENT / SHIPPED.
    Same citations-required check, but self-check now gives the model one
    retry with the specific rejection reason fed back before falling back
    to a deterministic refusal. See rag_pipeline.py MAX_ATTEMPTS,
    verify_answer, RETRY_SUFFIX_TEMPLATE.
Retrieval numbers are identical across all four runs (nothing about
retrieval/embeddings changed), so retrieval is listed once.

4. ROOT CAUSE: THE VACUOUS-PASS BUG (why Run B's faithfulness was inflated)
--------------------------------------------------------------------------------
Direct evidence (results/hallucination_eval.json under Run B, before this
fix): asking "What is the capital city of France?" got citations=[],
insufficient_context=False, and grounded=True. The three original
grounding checks (check_citation_pages, check_quotes_grounded,
check_numbers_grounded) all iterate over answer.citations or the answer's
numeric tokens -- with zero citations and zero numbers, each check has
nothing to inspect and returns no issues, so a fully fabricated answer
registered as "grounded" purely because nothing was ever actually
checked. Confirmed live (see conversation): the exact citations=[] /
grounded=True pairing was reproduced on demand for both the cookie-recipe
and capital-of-France questions.

Fix, in two layers:
  (a) hallucination.check_citations_required (deterministic, non-blocking
      detector, always on): if insufficient_context is False and
      citations is empty, that's now itself flagged as a grounding issue.
      This alone fixed the DETECTION problem and is why Run B's inflated
      0.929/1.000-style faithfulness numbers are no longer possible --
      grounding now honestly reflects whether evidence was ever provided.
  (b) rag_pipeline self-check loop (verify_answer + MAX_ATTEMPTS=2): fixes
      the RESPONSE problem -- rather than just detecting and discarding,
      the model gets fed the specific rejection reason and one chance to
      correct it (add a citation, or admit insufficient_context) before a
      deterministic refusal is used as the last resort. Trying (a) alone
      with an unconditional discard-on-reject policy (Run C) was measured
      to be actively harmful -- it discarded 100% of in-scope answers,
      including correct ones, because this 1.5B model frequently omits
      citations regardless of whether its answer content is right. The
      retry loop (Run D) is what makes the fix net-positive instead of
      just trading one failure mode (silent fabrication) for another
      (silent data loss).
